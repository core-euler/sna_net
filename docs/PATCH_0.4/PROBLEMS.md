1. Улучшение голосового ввода (длинные записи)

1.1 Проблема

При длинной записи голоса:
	•	анализ голоса завершается ошибкой
	•	распознавание не происходит
	•	пользователь теряет текст

При коротких записях (15–20 сек) всё работает стабильно.

⸻

1.2 Цель

Обеспечить возможность одной непрерывной длинной записи речи с корректным распознаванием.

⸻

1.3 Требования

Пользователь должен иметь возможность:
	•	записывать речь длительностью 60 секунд +
	•	получать единый текст
	•	не сталкиваться с ошибками при длинной записи

⸻

1.4 Решение

Реализовать скрытое чанкирование аудио:
	•	запись разбивается на сегменты (10–20 секунд)
	•	каждый сегмент отправляется отдельно на STT
	•	результаты объединяются в один текст

Важно:
	•	UX должен оставаться как «одна запись»
	•	пользователь не видит сегментацию

⸻

1.5 Поведение при повторной записи

Если пользователь снова включает запись:
	•	новый текст добавляется (append)
	•	предыдущий текст не удаляется

⸻

1.6 Надёжность

Добавить:
	•	retry при ошибке STT
	•	fallback при частичной ошибке (часть текста сохраняется)

⸻

1.7 Голосовой ввод в чате

голосовой ввод работает в разделе записи сна. 
Но не работает в Чате по сну - когда чат становится доступным после анализа
Кнопка хаписи голоса для создания сообщения в чате не срабатывает. 
Она должна работать точно так же как и в записи новых снов

⸻

2. Введение user.md (долгосрочная память LLM)

2.1 Цель

Создать внутреннюю сущность:

user.md

Это невидимый пользователю документ, который:
	•	агрегирует выводы LLM
	•	фиксирует динамику личности
	•	служит контекстом для анализа

⸻

2.2 Назначение

user.md — это:

“оперативная карта психики пользователя”

⸻

2.3 Содержимое

LLM должен записывать краткие тезисы:
	•	повторяющиеся темы
	•	доминирующие архетипы
	•	эмоциональные состояния
	•	внутренние конфликты
	•	направления изменений

Формат:

- recurring: control / chaos
- archetype: shadow (growing)
- emotional shift: anxiety → exploration
- phase: transition


⸻

2.4 Когда обновляется

user.md обновляется:
	•	после каждого анализа сна
	•	после значимых диалогов

⸻

2.5 Поведение обновления
	•	документ не перезаписывается полностью
	•	он эволюционирует
	•	устаревшие паттерны могут ослабляться

⸻

2.6 Использование в LLM

LLM обязан:
	•	читать user.md перед анализом
	•	учитывать динамику
	•	не дублировать уже известные выводы

⸻

3. Улучшение карты символов и архетипов

3.1 Проблема

Сейчас:
	•	каждый сон формирует отдельный кластер
	•	точки сильно локализованы
	•	почти нет пересечений

Это мешает видеть:
	•	связи между снами
	•	повторяющиеся символы
	•	глобальные паттерны

⸻

3.2 Цель

Создать единое смысловое пространство, где:
	•	элементы из разных снов пересекаются
	•	возникают неочевидные связи

⸻

3.3 Требования
	•	увеличить дисперсию точек
	•	убрать жёсткую привязку к отдельному сну
	•	усилить глобальные зависимости

⸻

3.4 Подход

Рекомендуемые изменения:
	•	нормализация embedding
	•	уменьшение веса “сна” как кластера
	•	глобальное позиционирование
	•	добавление стохастичности (noise)

⸻

3.5 Ожидаемый результат
	•	точки распределены по пространству
	•	появляются пересечения
	•	визуально видны паттерны

⸻

3.6 Зум карты

Текущий максимум:

x3

Требование:

x10


⸻

4. Временная динамика данных

4.1 Проблема

При работе с embedding:
	•	теряется временной контекст
	•	невозможно анализировать развитие

⸻

4.2 Цель

Добавить временную ось в систему анализа.

⸻

4.3 Требования

Каждый chunk должен содержать:

timestamp

Минимум:
	•	дата
	•	время

⸻

4.4 Структура

{
  text: ...
  embedding: ...
  created_at: datetime
}


⸻

4.5 Использование

LLM должен:
	•	учитывать порядок событий
	•	сравнивать прошлое и настоящее
	•	выявлять динамику изменений

⸻

4.6 Новые возможности

Система должна позволять:
	•	анализ эволюции психики
	•	поиск циклов
	•	выявление трендов

⸻


5. Acceptance Criteria

Патч считается завершённым, если:
	1.	Длинная аудиозапись (>30–60 сек) стабильно распознаётся.
	2.	Пользователь видит непрерывный UX записи.
	3.	Повторная запись дополняет текст.
	4.	user.md создаётся автоматически.
	5.	user.md обновляется после анализов.
	6.	LLM использует user.md как контекст.
	7.	Карта символов имеет высокую дисперсию.
	8.	Появляются пересечения между снами.
	9.	Зум увеличен до x10.
	10.	Каждый chunk содержит timestamp.
	11.	LLM учитывает временную динамику.
	12.	Все пользовательские строки в клиенте локализованы (нет хардкода на русском).
