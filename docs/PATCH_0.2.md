# JungAI — Patch Spec v1 (Client UI + Search + Archetypes + Dream metadata)
Дата: 2026-02-26  
Область: Flutter client + Backend (FastAPI)  
Цель: доработки по фидбэку тестеров. Документ для реализации патча.

---

## 0) Ключевая модель UX (как должно быть)
- Главный экран = **список снов (как список чатов в ChatGPT)**: `title + date/time + (опционально) gradient`.
- Внутри сна = экран/чат конкретного сна.
- Анализ сна делается **ровно 1 раз** на сон → сохраняется результат + метаданные (title, gradient, архетипы delta).
- “О себе” (about_me) сильно влияет на качество анализа → нужен онбординг при первом входе.

---

## 1) Онбординг “первый вход” (Client + Backend)

### 1.1 Как считать “первый вход”
- **Не по устройству**, а по пользователю: `user_id` в БД.
- Признак: у пользователя `onboarding_completed = false` (новое поле), либо вычисление “первый вход” как `about_me is null/empty` + `onboarding_seen=false`.

### 1.2 Поток онбординга (MVP: 2 шага)
1) Экран 1: **Приветствие**
   - Текст: что это за приложение, как им пользоваться.
   - Кнопка: `Далее`
   - Кнопка: `Пропустить` (опционально, но желательно)

2) Экран 2: **“О себе”**
   - Один `textarea` (многострочный ввод).
   - Подсказки + примеры (пояснить, зачем нужно: контекст личности для психоанализа).
   - Кнопки:
     - `Сохранить` (сохраняет on backend)
     - `Пропустить` (разрешено)

### 1.3 Требования
- Онбординг показывается **автоматически** после первого создания/получения user_id (после anonymous auth).
- После “Сохранить” или “Пропустить”:
  - поставить флаг `onboarding_completed = true` на бэке
  - больше не показывать автоматически.
- Пользователь всё равно может изменить “О себе” позже в профиле (существующий функционал сохраняется).

### 1.4 Backend изменения
- Добавить поле в `users`:
  - `onboarding_completed BOOLEAN NOT NULL DEFAULT false`
- Endpoint:
  - `PATCH /api/v1/users/me`
    - body: `{ "about_me": "...", "onboarding_completed": true }`
  - или отдельный:
    - `POST /api/v1/users/me/onboarding/complete`

## 2) Главный список снов — визуальные правки (Client)

### 2.1 Divider между элементами
- Между карточками/строками снов добавить тонкий разделитель `1–2px`.
- Divider должен учитывать тему:
  - Light: светло-серый
  - Dark: тёмно-серый / полупрозрачный

### 2.2 Перестановка даты/времени
Сейчас: дата слева, title справа.  
Нужно:
- **Title** — на всю ширину строки (в одну или две строки по дизайну)
- **Date + time** — маленьким шрифтом в **левом нижнем углу под title**
- Формат оставить текущий: `DD.MM.YYYY HH:mm` (или существующий формат проекта)

### 2.3 Title: 16 → 64
- В UI главного списка должен отображаться title до **64 символов** (c ellipsis при превышении).
- Внутри экрана сна title (в header) также должен быть 64.
- Логика title по умолчанию:
  - если пользователь не задавал title: взять первые N символов из `content` и сохранить как title
  - N = **64** (вместо 16)

### 2.4 Никаких новых экранов вместо списка
- Главный экран остаётся списком.
- Мы только улучшаем отображение.

## 3) Профиль — график + UI правки (Client)

### 3.1 График “сны” (14 дней вместо дней недели)
Сейчас: по дням недели.  
Нужно: **последние 14 календарных дней**, ось X = числа месяца (и/или `DD.MM`).

Требования:
- 14 столбиков/точек (в зависимости от текущего вида графика).
- Нулевые дни тоже показывать (0).
- Подписать значения по датам читаемо.

### 3.2 Горизонтальные полосы (grid) по Y
- Добавить **только горизонтальные** линии, параллельные X.
- Вертикальные линии не нужны.

### 3.3 Карточки статистики уменьшить по высоте
- Карточки “Всего снов” и “Streak” (и прочие) сделать ниже (уменьшить вертикальные padding/height).
- Цель: визуально компактнее, чтобы не “съедали экран”.

## 4) Toast/Popup уведомления — перенос вверх (Client)

### 4.1 Что есть
- Сейчас показываются всплывающие сообщения снизу (toast/snackbar-style) и перекрывают кнопки.

### 4.2 Что нужно
- Те же сообщения (успех/ошибка), та же форма/анимация, **но позиция сверху**.
- Появление: под статус-баром/под AppBar.
- Тайминг: оставить текущий (~2 сек).
- Перекрытие контента допустимо, но не должно мешать основным кнопкам ввода снизу.

## 5) Голосовой ввод (Client)

### 5.1 Длительность
- Сейчас запись обрывается ~через 3 секунды.
- Нужно: минимум **60 секунд** на одну сессию диктовки.

### 5.2 Повторное нажатие: append вместо replace
Требования:
- Если в поле ввода уже есть текст:
  - при новой диктовке результат **добавляется в конец** (append), а не стирает старое.
- Пользователь может:
  - надиктовать часть → поправить руками → снова надиктовать → текст дополняется.

### 5.3 UX управления
- Поведение микрофона:
  - нажал → запись началась
  - нажал ещё раз → остановить
  - таймаут 60 сек → авто-стоп (с результатом)
- Показать индикатор записи (мигающий микрофон/волна/таймер) — минимально.

## 6) Семантический поиск (Backend + Client)

### 6.1 Проблема
- Сейчас поиск строго по подстроке/точному слову, не понимает окончания/формы.

### 6.2 Решение: Embeddings + Vector search (pgvector)
- Ввести семантический поиск по embeddings:
  - хранить embedding для каждого сна (content/title) в БД
  - выполнять nearest-neighbor поиск по запросу пользователя
- Архитектура: “RAG-lite”
  - (MVP) embeddings → topK → вернуть результаты
  - (позже) embeddings + rerank (опционально)

### 6.3 Данные
- Для каждого Dream хранить:
  - `embedding VECTOR(d)` (d зависит от модели)
  - `embedding_model TEXT`
  - `embedding_updated_at TIMESTAMP`

### 6.4 Endpoints
- `GET /api/v1/dreams/search?q=...`
  - новый режим: `mode=semantic|lexical`
  - default = `semantic` (или переключатель в UI)
- Response: список dream ids + score + snippet.

### 6.5 Пересчёт embeddings
- При создании сна → посчитать embedding async (Celery task).
- При редактировании сна → пересчитать embedding.

## 7) Изменение даты записи сна (Client + Backend)

### 7.1 Требование
- Пользователь должен уметь менять `created_at` (дату/время записи) вручную.

### 7.2 Где доступно
1) В главном списке: **long press** по карточке сна → меню → “Изменить дату”
2) Внутри сна: меню “…” → “Изменить дату”

### 7.3 UI
- Открыть date-time picker:
  - выбор даты + времени
- Ограничения:
  - нельзя будущие даты (по умолчанию запретить)
  - разрешить прошлые даты без лимита (или минимум с 2020)

### 7.4 Backend
- `PATCH /api/v1/dreams/{id}`
  - добавить поле `created_at` (если сейчас запрещено)
  - валидация: `created_at <= now()`

## 8) Архетипы (Backend + LLM + Client Profile)

### 8.1 Суть
- Только Jungian анализ (другие режимы отключены).
- На каждый сон анализ делается 1 раз.
- При анализе LLM должна вернуть:
  1) текст анализа
  2) JSON delta архетипов: какие архетипы встретились → `+1`

### 8.2 Формат JSON delta
Пример:
```json
{
  "archetypes_delta": {
    "Тень": 1,
    "Анима": 1
  }
}

Правила:
	•	Значения только положительные целые (в MVP всегда 1).
	•	Архетипы динамические: если новый — добавить в словарь пользователя.

8.3 Хранение на backend

Персональный словарь на пользователя:
	•	Таблица user_archetypes:
	•	user_id
	•	name TEXT
	•	count INT
	•	UNIQUE(user_id, name)

Алгоритм обновления:
	•	Для каждого (name, +1):
	•	upsert: count = count + delta

8.4 Отображение в профиле
	•	Под графиком “сны за 14 дней” добавить блок “Архетипы”.
	•	Визуально: горизонтальные бары (progress bars / bar chart horizontal).
	•	Показывать Top-10 архетипов по count (по убыванию).
	•	Для каждого: name + count.

8.5 Стабильность
	•	Архетипы обновляются только при первом анализе сна.
	•	Диалог после анализа не влияет на архетипы.

9) Авто-title + градиент цвета сна после анализа (LLM + Backend + Client)

9.1 Title после анализа
	•	После получения анализа LLM должна вернуть осмысленный title.
	•	Backend сохраняет title в Dream.
	•	Client обновляет список снов (title меняется автоматически).

9.2 Градиент (2 цвета + переход в фон темы)
	•	LLM возвращает два HEX цвета:
	•	#RRGGBB и #RRGGBB
	•	Backend сохраняет их в Dream:
	•	gradient_color_1
	•	gradient_color_2
	•	Client в списке снов рисует градиент:
	•	слева направо: color1 -> color2 -> themeBackground
	•	ширина градиентной зоны: примерно 30% карточки слева
	•	дальше фон карточки = фон темы (white / dark)

9.3 Когда градиент применяется
	•	Только после анализа (когда анализ сохранён).
	•	До анализа: градиента нет (карточка обычная).

9.4 Формат ответа анализа (LLM service → backend)

LLM должен возвращать структуру:

{
  "analysis_text": "....",
  "title": "Осмысленное название сна (до 64)",
  "gradient": {
    "color1": "#AABBCC",
    "color2": "#DDEEFF"
  },
  "archetypes_delta": {
    "Тень": 1,
    "Самость": 1
  }
}

Backend:
	•	сохраняет analysis_text (как сейчас)
	•	обновляет dream.title
	•	обновляет dream.gradient_color_1/2
	•	применяет archetypes_delta в user_archetypes
