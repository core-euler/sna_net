# JungAI — Patch Spec: Automatic Dream Analysis After Save

Версия: 1.0
Статус: ТЗ для патча
Область: клиент + backend interaction
Важно: этот документ описывает **только** автоматический запуск анализа после сохранения сна, loading-state и retry-логику при обращении к LLM.

---

## 1. Цель

Убрать ручной запуск анализа сна из пользовательского сценария.

Новая логика:

* пользователь создаёт сон;
* сон сохраняется;
* анализ запускается автоматически сразу после сохранения;
* пользователь видит loading-state, пока анализ не завершён;
* при временных ошибках LLM используются автоматические retry-попытки;
* при окончательном fail пользователь видит состояние ошибки и может вручную повторить попытку.

---

## 2. Что меняется в UX

### 2.1 Было

Сейчас анализ запускается вручную:

* кнопкой `Анализировать` внутри сна;
* или через соответствующий пункт во всплывающем меню / меню опций.

### 2.2 Должно стать

После сохранения сна анализ должен запускаться автоматически.

Новая последовательность:

1. Пользователь записывает сон.
2. Нажимает `Отправить` / `Сохранить`.
3. Сон сохраняется в БД.
4. Backend автоматически создаёт задачу анализа.
5. Клиент показывает loading-state.
6. После завершения анализа карточка сна и экран сна обновляются.

---

## 3. Изменения в UI

### 3.1 Удалить ручной запуск анализа

Из пользовательского интерфейса нужно убрать:

* кнопку `Анализировать` внутри сна;
* пункт `Анализировать` из контекстного меню / всплывающего окна;
* любую механику, где пользователь отдельно инициирует обычный анализ нового сна.

Важно:

* ручной `retry` после окончательной ошибки допускается;
* но базовый сценарий анализа должен быть полностью автоматическим.

---

## 4. Loading-state во время анализа

### 4.1 Требование

Пока анализ выполняется, пользователь должен видеть стандартную анимацию загрузки.

Использовать:

* обычный вращающийся loading spinner;
* визуально ориентироваться на стандартный круговой индикатор загрузки.

### 4.2 Где отображать

Loading-state должен поддерживаться в двух местах:

#### A. На главном экране (плитка сна)

Если пользователь находится на grid-экране со снами:

* карточка только что созданного сна должна перейти в состояние `анализируется`;
* поверх карточки или внутри неё показывается spinner;
* title / gradient могут быть временно placeholder-состоянием, пока анализ не завершён.

#### B. Внутри экрана сна

Если пользователь сразу открыл сон:

* на экране сна также показывается loading spinner;
* после завершения анализа UI обновляется автоматически.

### 4.3 Поведение loading-state

Минимально допустимое поведение:

* сон сохраняется сразу;
* UI не зависает;
* пока анализ идёт — отображается состояние `analyzing`;
* после завершения анализированная сущность обновляется автоматически.

---

## 5. Новые статусы сна

У сна на клиенте и/или на backend должен быть понятный статус обработки.

Минимальный набор статусов:

* `saved` — сон сохранён, анализ ещё не стартовал или не отражён в UI
* `analyzing` — анализ выполняется
* `analyzed` — анализ завершён успешно
* `analysis_failed` — анализ завершился ошибкой после всех retry

Если в проекте уже есть task-status model, использовать её. Если нет — добавить явное поле статуса.

---

## 6. Retry-логика при обращении к LLM

### 6.1 Требование

При временной ошибке вызова LLM анализ не должен падать сразу после первой неудачной попытки.

Обязательное поведение:

* выполнить несколько автоматических retry-попыток;
* между попытками должна быть задержка;
* retry выполняется на backend / worker side, а не на клиенте.

### 6.2 Где реализовать

Retry должен быть реализован:

* в backend processing pipeline;
* в задаче фоновой обработки анализа;
* не в Flutter-клиенте.

Клиент только отображает статус и результат.

### 6.3 Рекомендуемая стратегия retry

Рекомендуемый вариант:

* 3–5 попыток;
* небольшая задержка между попытками;
* желательно exponential backoff или линейный backoff;
* логировать каждую неудачу.

### 6.4 Ошибки, которые должны идти в retry

Retry нужен для временных ошибок, например:

* timeout;
* временная недоступность LLM;
* сетевой сбой;
* rate limit / transient provider error.

### 6.5 Ошибки, которые не должны бесконечно повторяться

Если ошибка заведомо неустранимая:

* invalid request;
* malformed payload;
* hard validation error;

то задача должна завершаться fail-state без бесконечных повторов.

---

## 7. Fail-state после окончательной ошибки

Если все retry-попытки исчерпаны и анализ всё равно не выполнен:

* сон остаётся сохранённым;
* статус сна становится `analysis_failed`;
* клиент показывает ошибку анализа;
* пользователь должен иметь возможность вручную нажать `Повторить анализ`.

Важно:

* это **не возвращает старую механику ручного запуска анализа**;
* это только fallback для редкого случая окончательного fail.

---

## 8. Что должно обновляться после успешного анализа

После успешного завершения автоматического анализа клиент должен получить обновлённые данные сна.

Минимум:

* `analysis_text`
* `title` (если он пересчитывается анализом)
* `gradient` (если он задаётся анализом)
* `analysis status = analyzed`

Карточка сна на главном экране должна обновиться автоматически без ручного refresh со стороны пользователя.

---

## 9. Backend requirements

### 9.1 Автоматический запуск анализа

После успешного создания сна backend должен автоматически инициировать задачу анализа.

Варианты реализации:

* через существующий Celery worker;
* через существующую очередь фоновых задач.

### 9.2 Статус задачи

Backend должен уметь вернуть клиенту статус анализа.

Если task-status endpoint уже существует — использовать его.
Если нет — добавить понятный механизм проверки статуса.

### 9.3 Идемпотентность

Если пользователь случайно повторил действие сохранения или клиент повторно запросил статус:

* не должно создаваться несколько параллельных анализов для одного и того же сна.

Правило:

* один сон → один основной анализ;
* повторная попытка допустима только в случае fail-state.

---

## 10. Client requirements

### 10.1 После создания сна

После успешного ответа на создание сна клиент должен:

1. добавить сон в локальный список / grid;
2. сразу отобразить его как `analyzing`;
3. показать spinner;
4. подписаться на обновление статуса или периодически запрашивать его;
5. после завершения заменить loading-state на нормальную карточку.

### 10.2 При ошибке анализа

Если backend вернул `analysis_failed`:

* показать fail-state;
* показать понятную ошибку пользователю;
* дать кнопку `Повторить анализ`.

---

## 11. Что не входит в этот патч

Не входит в scope:

* redesign grid / плитки как отдельная задача;
* изменение RAG;
* изменение retrieval;
* изменение архетипов;
* изменение поиска;
* изменение структуры gradient logic;
* новые режимы анализа.

Этот документ касается только:

* автоматического старта анализа;
* loading-state;
* retry-логики;
* fail-state и ручного retry.

---

## 12. Acceptance Criteria

Патч считается завершённым, если:

1. После сохранения сна анализ стартует автоматически.
2. Пользователь больше не запускает анализ вручную в обычном сценарии.
3. Кнопка `Анализировать` и соответствующий пункт меню удалены из UI.
4. Во время анализа отображается spinner.
5. Loading-state виден и в карточке сна, и/или на экране самого сна.
6. При временной ошибке LLM backend делает автоматические retry-попытки.
7. После успешного анализа карточка сна автоматически обновляется.
8. После окончательной ошибки сон получает fail-state.
9. В fail-state у пользователя есть кнопка `Повторить анализ`.
10. Для одного сна не создаются дублирующиеся параллельные анализы.
