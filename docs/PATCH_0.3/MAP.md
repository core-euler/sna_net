# ТЗ — Dream Sphere: визуализация векторного пространства снов
**Проект:** JungCore | **Версия:** 1.0 | **Стек:** FastAPI + Flutter

---

## 1. Обзор и цели

Модуль визуализирует RAG-базу снов пользователя в виде интерактивной 3D-сферы. Каждый чанк (фрагмент записи сна) — точка на поверхности сферы. Близкие по смыслу чанки расположены рядом, кластеры соответствуют юнгианским архетипам.

**Зачем это нужно:**
- Пользователь видит целостную карту своего «семантического ландшафта» снов
- Обнаруживает повторяющиеся архетипические паттерны визуально, без чтения
- Может тапнуть на любой чанк и перейти к записи сна
- WOW-экран для онбординга и удержания

### 3.2 API-эндпоинты

#### `GET /api/v1/sphere/{user_id}`

Основной эндпоинт. Возвращает список чанков пользователя с 3D-координатами на сфере, цветами архетипов и метаданными.

**Query-параметры:**

| Параметр | Тип | По умолчанию | Описание |
|---|---|---|---|
| n_neighbors | int | 15 | UMAP: баланс локальной / глобальной структуры |
| min_dist | float | 0.1 | UMAP: плотность расположения точек в кластере |
| cluster_method | str | hdbscan | Метод кластеризации |
| force_refresh | bool | false | Сбросить Redis-кеш и пересчитать |

**Структура ответа — каждый узел содержит:**
- `id` — идентификатор чанка
- `x, y, z` — координаты на единичной сфере, каждое в диапазоне [-1, 1]
- `cluster_id` — к какому кластеру принадлежит, -1 означает шум
- `cluster_label` — название архетипа: Тень, Анима, Самость и т.д.
- `archetype_color` — hex-цвет кластера для рендера
- `cosine_sim_to_center` — насколько чанк типичен для своего кластера
- `text_preview` — первые 80 символов текста сна
- `date` — дата записи
- `emotion_valence` — эмоциональный тон от -1.0 (тёмный) до +1.0 (светлый)
- `tokens` — длина чанка в токенах

**Структура ответа — каждый кластер содержит:**
- `id, label, color` — идентификатор, название, цвет
- `count` — количество чанков в кластере
- `center` — центроид кластера в 3D-пространстве

---

#### `GET /api/v1/sphere/{user_id}/chunk/{chunk_id}`

Возвращает полный текст чанка и список ближайших соседей по косинусной близости. Вызывается при тапе на точку — наполняет ChunkDetailSheet.

---

#### `WebSocket /api/v1/sphere/{user_id}/stream`

Используется, когда у пользователя более 200 чанков. Сервер отправляет узлы партиями по 20 штук с указанием прогресса. Flutter строит сферу прогрессивно, не блокируя UI.

```
Клиент подключается
     │
     ├─→ batch: [20 nodes], progress: 0.1
     ├─→ batch: [20 nodes], progress: 0.2
     │   ...
     └─→ complete: { clusters: [...] }
```

### 3.3 Логика редукции: от эмбеддингов к сфере

Весь процесс — четыре последовательных шага:

```
Шаг 1. Извлечение
  ChromaDB → все эмбеддинги пользователя
  Форма данных: матрица (N чанков × 1536 измерений)
        │
        ▼
Шаг 2. UMAP-редукция
  1536 измерений → 3 измерения
  Сохраняет топологию: близкие смыслы остаются рядом
  Форма данных: матрица (N × 3)
        │
        ▼
Шаг 3. Нормализация на сферу
  Каждый вектор делится на свою длину
  Все точки оказываются на поверхности единичной сферы
  Форма данных: матрица (N × 3), длина каждой строки = 1
        │
        ▼
Шаг 4. Кластеризация
  HDBSCAN находит группы в 3D-пространстве
  Каждой группе присваивается лейбл архетипа
```

**Почему UMAP, а не t-SNE:**
- UMAP сохраняет глобальную структуру — кластеры архетипов разнесены в пространстве, не склеятся
- t-SNE нестабилен: при разных запусках кластеры могут менять расположение
- UMAP значительно быстрее при 500+ чанков
- Параметр `metric='cosine'` — стандарт для NLP-эмбеддингов; евклидова метрика даёт хуже кластеры

**Что теряет нормализация на сферу:**
Радиальная компонента — насколько чанк «типичен» для своего архетипа. Альтернатива: не нормализовать, тогда центр пространства = усреднённый архетип, а периферия = аномальные сны. Для JungCore нормализация предпочтительна: акцент на отношениях между чанками, а не на их «весе».

### 3.4 Кеширование

UMAP — медленная операция: от 2 до 10 секунд для 200+ чанков. Результат сохраняется в Redis.

**Ключ кеша** строится из `user_id` + хеша параметров UMAP (`n_neighbors`, `min_dist`). Это позволяет хранить несколько проекций одного пользователя с разными параметрами.

**Инвалидация:** при добавлении нового чанка сна все ключи пользователя удаляются. Следующий запрос пересчитывает проекцию.

**TTL:** 1 час — компромисс между свежестью данных и нагрузкой на сервер.

---

### 3.5 Маппинг архетипов

После кластеризации каждому кластеру присваивается название юнгианского архетипа. Два подхода:

**Подход A — Seed-векторы (рекомендуется):**
Заранее эмбеддируем текстовые описания 12 архетипов. Для центроида каждого кластера считаем косинусную близость ко всем seed-векторам. Наиболее близкий архетип → лейбл кластера. Быстро, детерминировано, работает офлайн.

**Подход B — LLM-классификация (fallback):**
Берём 3 случайных чанка из кластера и отправляем в LLM с промптом на определение доминирующего архетипа. Гибче, но дороже и медленнее.

### 4.3 Математика 3D-рендера

Flutter не имеет встроенного 3D-движка. Сфера рендерится вручную через `CustomPainter` с перспективной проекцией.

**Принцип работы каждого кадра:**

```
Каждый кадр (60 FPS):
  1. Получить текущие углы вращения rotX, rotY
  2. Для каждой точки (x, y, z):
       a. Применить вращение по Y (горизонталь)
       b. Применить вращение по X (вертикаль)
       c. Перспективная проекция → экранные координаты (sx, sy)
       d. Сохранить projectedZ для z-сортировки
  3. Отсортировать точки по projectedZ: дальние рисуются первыми
  4. Нарисовать wireframe сферы
  5. Нарисовать точки с учётом глубины и opacity
```

**Перспективная проекция:** чем дальше точка (меньше Z), тем она меньше. Близкие точки крупнее. Точки на обратной стороне сферы (Z < 0) рисуются с opacity 15% — эффект «просвечивания».

**Wireframe сферы:** 12 меридианов + 5 параллелей, нарисованных теми же матрицами вращения. Создаёт ощущение объёма без 3D-движка.

### 4.4 Hit-testing — определение тапа

```
Пользователь тапает → GestureDetector.onTapUp → (tapX, tapY)
     │
     ▼
Итерация по точкам от ближних к дальним (обратный z-order)
     │
     ├── Пропустить точки на обратной стороне (projectedZ < -0.3)
     │
     └── Для каждой точки: distance(screenPos, tap)
               │
               ├── distance <= hitRadius (16-20px) → ВЫБРАНА
               │
               └── продолжить
```

Радиус хита (16-20px) намеренно больше видимого размера точки — компенсирует погрешность тапа на мобильном экране.

## 5. Модели данных

### Структура ChunkNode

```
ChunkNode
  ├── id: string               — уникальный идентификатор
  ├── position                 — координаты на сфере
  │     ├── x: float [-1, 1]
  │     ├── y: float [-1, 1]
  │     └── z: float [-1, 1]
  ├── clusterId: int           — -1 если шум
  ├── clusterLabel: string     — название архетипа
  ├── color: Color             — цвет архетипа
  ├── cosSimToCenter: float    — типичность для кластера [0, 1]
  ├── textPreview: string      — первые 80 символов
  ├── date: DateTime           — дата записи сна
  ├── emotionValence: float    — тон [-1.0 тёмный .. +1.0 светлый]
  └── tokens: int              — длина чанка
```

### Runtime-поля (обновляются каждый кадр, не из API)

```
ChunkNode (runtime)
  ├── screenPos: Offset        — экранные координаты после проекции
  ├── projectedZ: float        — глубина для z-сортировки и fade
  └── hitRadius: float         — зона тапа в пикселях
```

## 9. Открытые вопросы

**ChromaDB vs Qdrant** — ChromaDB проще в старте, Qdrant лучше справляется при 10k+ чанков и имеет более богатый API фильтрации. Решение зависит от прогноза роста JungCore.

**Seed-векторы vs LLM-классификация** — нужно протестировать точность на реальных данных пользователей. Seed-векторы быстрее и дешевле, но могут ошибаться на нетипичных снах.

**CustomPainter vs WebGL** — при N > 1000 чанков Flutter CustomPainter может не держать 60 FPS. Резервный вариант: flutter_gl или WebView с Three.js-рендером.

**Частота пересчёта UMAP** — пересчитывать при каждом новом сне (актуально, но дорого) или батчами раз в день (дешевле, но с задержкой отображения). ОТВЕТ - РАЗ В ДЕНЬ
