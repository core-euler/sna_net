# JungAI — RAG MVP: переход от отдельных снов к семантической памяти пользователя

Версия: 1.0
Статус: архитектурное ТЗ для разработки
Цель: перевести систему с модели анализа отдельных снов на модель персональной семантической памяти пользователя, где основная единица retrieval — смысловой chunk, а не сон целиком.

---

## 1. Продуктовая идея

Текущая модель работает с отдельными снами как с изолированными записями. Это ограничивает качество анализа, потому что повторяющиеся мотивы, символы, персонажи, места и смысловые пересечения между снами во времени остаются слабо заметны.

Новая модель должна строиться как персональная память пользователя:

* каждый сон становится источником смысловых фрагментов;
* фрагменты индексируются в векторной базе;
* из каждого сна извлекаются структурированные сущности;
* новые анализы опираются не только на текущий сон, но и на релевантный материал из всей накопленной истории.

Итоговая цель — не просто хранить сны, а формировать семантическое поле пользователя, в котором становятся видны:

* повторяющиеся паттерны;
* устойчивые мотивы;
* пересечения смыслов;
* динамика архетипов и символов во времени.

---

## 2. Цели RAG MVP

Первая версия RAG должна уметь:

1. Искать похожие паттерны и мотивы в прошлых снах пользователя.
2. Находить пересечения смыслов, мест, объектов и персонажей.
3. Использовать найденные chunk'и как контекст для нового анализа сна.
4. Извлекать и сохранять структурированные сущности MVP-уровня:

   * символы;
   * архетипы.

На этом этапе система не должна пытаться строить полноценный граф бессознательного или сложную временную аналитику. MVP должен доказать, что chunk-based retrieval реально улучшает качество анализа.

---

## 3. Основная единица retrieval

### Решение

Основная единица retrieval: **semantic chunk**.

### Почему не dream целиком

Сон целиком слишком грубая единица. В одном сне может быть несколько разных линий, образов, конфликтов и сцен, которые плохо ищутся как единый документ.

Пример:

* фрагмент 1: старый дом и вода в подвале;
* фрагмент 2: разговор с матерью;
* фрагмент 3: бегство от неизвестной фигуры.

Если хранить embedding только на весь сон, retrieval будет неточным. Если хранить embedding по chunk'ам, можно отдельно вытащить:

* мотив затопления;
* повторение материнской фигуры;
* повторяющийся сюжет преследования.

### Вывод

Векторизуются и ищутся именно chunk'и. Сон остается контейнером, но не главной retrieval-единицей.

---

## 4. Архитектурная модель памяти

Система должна состоять из четырех логических слоев.

### 4.1 Raw layer

Хранение исходных данных:

* сон как документ;
* дата записи;
* title;
* анализ;
* профиль пользователя (about_me);
* служебные поля.

Это обычный слой данных, на котором работает CRUD и UI.

### 4.2 Chunk layer

Каждый сон разбивается на semantic chunks.

Chunk — это смысловой фрагмент сна, который можно интерпретировать отдельно:

* сцена;
* эпизод;
* значимый образ;
* переход между состояниями;
* мини-сюжет.

Именно chunk:

* получает embedding;
* попадает в vector DB;
* участвует в retrieval.

### 4.3 Structured layer

Из сна и/или из chunk'ов извлекаются структурированные сущности MVP:

* символы;
* архетипы.

Дополнительно уже на первом этапе retrieval должен уметь учитывать пересечения по:

* местам;
* объектам;
* персонажам;
* мотивам.

Но хранить их как отдельные строгие сущности в MVP необязательно, если это усложняет систему. Для MVP допустимо:

* явно сохранять символы и архетипы;
* места/объекты/персонажи использовать в retrieval и summaries, а затем при необходимости расширить схему.

### 4.4 Reasoning layer

Этот слой формирует контекст для анализа:

* берет текущий сон;
* разбивает его на chunk'и;
* ищет похожие chunk'и в истории пользователя;
* собирает краткий контекст повторов и смысловых пересечений;
* передает все это LLM для финального анализа.

---

## 5. Новая продуктовая логика анализа

### Было

* пользователь записывает сон;
* система анализирует только этот сон;
* результат локален и почти не опирается на прошлые сны.

### Должно стать

* пользователь записывает сон;
* сон разбивается на semantic chunks;
* для каждого chunk выполняется retrieval по истории пользователя;
* система находит:

  * похожие паттерны;
  * смысловые пересечения;
  * общие объекты;
  * общие места;
  * повторяющихся персонажей;
  * повторение архетипов и символов;
* LLM пишет анализ уже не только по одному сну, а по текущему сну в контексте накопленной памяти.

---

## 6. Что именно должна уметь первая RAG-версия

### 6.1 Retrieval задачи

RAG MVP должен уметь извлекать из базы пользователя:

* chunk'и, похожие по смыслу;
* chunk'и, где встречаются похожие сцены или мотивы;
* chunk'и с пересечением по объектам, местам и персонажам;
* chunk'и, связанные с теми же символами;
* chunk'и, где уже проявлялись те же архетипы.

### 6.2 Analysis задачи

При анализе нового сна система должна уметь:

* выявить, какие мотивы уже повторялись раньше;
* показать, какие символы возвращаются из других снов;
* найти скрытые связи между внешне непохожими снами;
* использовать эти связи как контекст для объяснения сна.

### 6.3 Ограничения MVP

На первом этапе не требуется:

* строить полный knowledge graph;
* делать агентный диалог с памятью как с самостоятельной личностью;
* считать сложную временную динамику по всем сущностям;
* строить автоматические long-term summaries по месяцам и годам.

Это следующий этап. Сейчас задача — качественный chunk retrieval и контекстный анализ.

## 7. Data model — новые сущности

### 7.1 Dreams

Основная запись сна.

Минимальные поля:

* `id`
* `user_id`
* `title`
* `content`
* `analysis_text`
* `created_at`
* `updated_at`
* `gradient_color_1` (если уже используется)
* `gradient_color_2`

### 7.2 DreamChunks

Новая таблица / коллекция.

Поля:

* `id`
* `dream_id`
* `user_id`
* `chunk_index`
* `text`
* `embedding`
* `created_at`
* `metadata_json` — опционально

Назначение:

* хранение chunk'ов сна;
* векторный поиск;
* retrieval для анализа.

### 7.3 DreamSymbols

Новая таблица символов по сну или chunk'ам.

Поля:

* `id`
* `user_id`
* `dream_id`
* `chunk_id` nullable
* `symbol_name`
* `weight` nullable
* `created_at`

### 7.4 UserArchetypes

Таблица персонального словаря архетипов пользователя.

Поля:

* `id`
* `user_id`
* `archetype_name`
* `count`
* `updated_at`

Логика:

* после анализа каждого нового сна LLM возвращает `archetypes_delta`;
* backend увеличивает `count` соответствующих архетипов;
* если архетип новый, запись создается.

### 7.5 DreamArchetypes

Связь архетипов с конкретным сном.

Поля:

* `id`
* `user_id`
* `dream_id`
* `archetype_name`
* `delta`
* `created_at`

Это нужно для трассируемости: чтобы понимать, какой сон добавил какой архетип.

---

## 8. Векторная инфраструктура

### 8.1 Решение

Использовать **PostgreSQL + pgvector**.

### 8.2 Почему

* уже есть PostgreSQL в проекте;
* меньше инфраструктурной сложности, чем поднимать отдельную vector DB;
* подходит для MVP;
* легко сочетается с текущим backend stack.

### 8.3 Что векторизуем

На MVP обязательно векторизуются:

* semantic chunks.

Опционально, если хочется улучшить retrieval:

* title сна;
* analysis_text;
* full dream embedding как fallback.

Но базовая retrieval единица — chunk.

---

## 9. Chunking strategy

### 9.1 Требование

Нельзя делать произвольную нарезку по символам. Нужен смысловой chunking.

### 9.2 Допустимые стратегии MVP

Вариант A — LLM-based chunking:

* LLM получает сон;
* делит его на смысловые сцены/фрагменты;
* backend сохраняет chunk'и.

Вариант B — rule-based + fallback:

* сначала деление по абзацам / длинным паузам / признакам смены сцены;
* затем при необходимости LLM нормализует chunk'и.

### 9.3 Требование к chunk'у

Каждый chunk должен быть:

* семантически цельным;
* достаточно коротким для хорошего embedding;
* достаточно насыщенным, чтобы передавать сцену/образ/смысл.

---

## 10. Extraction pipeline

После создания сна запускается pipeline.

### Шаг 1. Сохранение raw dream

Сон сохраняется как обычная запись.

### Шаг 2. Chunking

Сон разбивается на semantic chunks.

### Шаг 3. Embeddings

Для каждого chunk считается embedding.

### Шаг 4. Extraction

Из сна и/или chunk'ов извлекаются:

* символы;
* архетипы.

### Шаг 5. Persist

Сохраняются:

* chunk'и;
* embeddings;
* symbols;
* dream_archetypes;
* обновления user_archetypes.

### Шаг 6. Analysis context build

Перед генерацией финального анализа выполняется retrieval:

* top-k похожих chunk'ов по embedding;
* top-k совпадений по символам;
* архетипические пересечения.

### Шаг 7. Final LLM analysis

LLM получает:

* текущий сон;
* текущие chunk'и;
* релевантные найденные chunk'и из истории;
* about_me пользователя;
* символы;
* архетипический контекст.

И на основе этого пишет анализ.

---

## 11. Что должно входить в retrieval context

Для каждого нового анализа система должна собирать контекст не как набор полных снов, а как набор релевантных фрагментов.

Минимальный контекст RAG MVP:

* текущий сон;
* top-k похожих chunk'ов по embedding;
* краткий список совпавших символов;
* краткий список архетипов пользователя, связанных с текущим сном;
* служебная summary-подсказка для LLM о найденных пересечениях.

Пример смыслов, которые retrieval должен уметь подсвечивать:

* повторяющееся место;
* повторяющийся персонаж;
* повторяющийся объект;
* одинаковый эмоциональный паттерн;
* похожая сцена перехода, преследования, потери контроля и т.д.

---

## 12. Что именно извлекать в MVP

### 12.1 Символы

В первую очередь сохраняем символы.

Примеры:

* вода
* дом
* лестница
* зеркало
* тень
* лес
* дверь

Требования:

* символы должны быть нормализованы;
* желательно приводить к единой словарной форме;
* дубликаты внутри одного сна схлопывать или снижать вес.

### 12.2 Архетипы

Вторая обязательная сущность MVP.

Примеры:

* Тень
* Самость
* Анима
* Анимус
* Мудрый старец
* Великая мать

Требования:

* архетипы динамические, не жестко зашитые;
* если LLM выявляет новый архетип, он добавляется в словарь пользователя;
* при анализе сна архетипы возвращаются как delta.

---

## 13. Какие пересечения должен находить retrieval

MVP retrieval должен быть заточен не только под “похожие тексты”, а под смысловые связи.

Нужно уметь поднимать:

* общие мотивы;
* общие места;
* общие объекты;
* общих персонажей;
* символические повторения;
* архетипические повторения.

Это значит, что scoring retrieval желательно строить как гибрид:

1. embedding similarity;
2. symbol overlap;
3. archetype overlap.

Даже если сначала это будет реализовано без сложного ранжирования, структура системы должна предусматривать такой гибрид.

## 14. Изменения в backend API

### 14.1 Новый pipeline после создания сна

После `POST /dreams` backend должен запускать асинхронную обработку:

* chunking;
* embeddings;
* extraction;
* analysis context build;
* final analysis generation.

### 14.2 Новые внутренние сущности ответа LLM

LLM/processing service должен возвращать структурированный JSON примерно такого вида:

```json
{
  "chunks": [
    {
      "index": 0,
      "text": "..."
    }
  ],
  "symbols": ["вода", "дом", "лестница"],
  "archetypes_delta": {
    "Тень": 1,
    "Самость": 1
  },
  "analysis_text": "..."
}
```

Примечание:

* title и gradient можно сохранять как отдельные поля, если они уже есть в текущем пайплайне.

### 14.3 Новый retrieval endpoint (опционально для debug/admin)

Желательно предусмотреть внутренний endpoint или debug mode:

* получить top-k chunk'ов, найденных для сна;
* посмотреть, почему именно они выбраны.

Это сильно упростит отладку RAG.

---

## 15. Изменения в клиенте

На первом этапе клиент не должен становиться тяжелым. Основные изменения происходят на backend и в analysis pipeline.

### Client MVP задачи

1. Ничего не ломать в текущем UX создания сна.
2. После анализа уметь получать обновлённые данные сна:

   * title;
   * analysis_text;
   * градиент;
   * при необходимости флаг “анализ выполнен”.
3. В профиле продолжать отображать архетипы, но теперь они будут строиться на более надёжной семантической основе.

Прямой chunk retrieval UI в MVP не обязателен.

---

## 16. Нефункциональные требования

### 16.1 Масштабируемость

Система должна быть рассчитана на большое число снов у одного пользователя.

### 16.2 Асинхронность

Chunking, embeddings, extraction и analysis должны выполняться в фоне.

### 16.3 Идемпотентность

Повторный запуск processing pipeline для одного сна не должен дублировать chunk'и и архетипы.

### 16.4 Наблюдаемость

Нужны логи по этапам:

* dream saved;
* chunking done;
* embeddings done;
* extraction done;
* retrieval done;
* analysis saved.

## 17. Этапы внедрения

### Этап 1 — базовый RAG MVP

Сделать:

* таблицу dream_chunks;
* pgvector;
* embeddings для chunk'ов;
* retrieval по chunk'ам;
* extraction символов и архетипов;
* LLM analysis с учетом retrieval context.

### Этап 2 — улучшение качества retrieval

Добавить:

* гибридный scoring;
* нормализацию символов;
* debug/trace retrieval.

### Этап 3 — расширение памяти

Позже можно добавить:

* временную динамику;
* monthly/yearly summaries;
* graph of motifs;
* агентный диалог с персональной памятью пользователя.

---

## 18. Acceptance criteria

RAG MVP считается внедрённым, если:

1. После сохранения сна он разбивается на semantic chunks.
2. Для chunk'ов создаются embeddings и они сохраняются в pgvector.
3. Для сна извлекаются символы и архетипы.
4. Новый анализ опирается не только на текущий сон, но и на найденные релевантные chunk'и из истории пользователя.
5. Система умеет находить и учитывать:

   * повторяющиеся паттерны;
   * пересечения по местам;
   * пересечения по объектам;
   * пересечения по персонажам;
   * повторяющиеся символы и архетипы.
6. Архетипы пользователя обновляются через delta-механику.
7. Вся система работает на PostgreSQL + pgvector без отдельной внешней vector DB.

---

## 19. Итоговая концептуальная формула

JungAI должен перейти от модели “анализ отдельного сна” к модели “персональная семантическая память пользователя”.

Сон — это не конечная единица смысла.
Сон — это источник фрагментов, символов и архетипических сигналов.
Retrieval должен работать по chunk'ам.
Анализ должен опираться на общее поле смыслов пользователя, а не только на один документ.

Именно это является стратегическим направлением RAG MVP.
