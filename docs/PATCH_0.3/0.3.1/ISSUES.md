# PATCH 0.3.1 — Issues

## 1. Чат по сну не работает

**Симптом:** После получения анализа пользователь отправляет follow-up сообщение в чат — ответ никогда не приходит.

**Причина:** Celery-задача `reply_to_dream_chat_task` не имеет конфигурации ретраев (`autoretry_for`, `max_retries` и т.д.). При любой ошибке от LLM-сервиса задача падает навсегда с первой попытки. В логах видно, что Gonka Proxy возвращает 500 из-за таймаута к апстриму (`i/o timeout` к `85.234.91.100:8000`). LLM-сервис пробрасывает 500 в бэкенд, бэкенд бросает `LLMTransientError`, но задача чата не настроена на ретраи — и просто падает.

**Дополнительная проблема:** У задачи чата нет механизма обновления статуса сообщения при ошибке. Клиент не получает информации о том, что генерация ответа провалилась, и сообщение зависает в состоянии "в процессе" навсегда.

**Решение:**
- Добавить в `reply_to_dream_chat_task` конфигурацию ретраев аналогично `analyze_dream_task`: `autoretry_for=(LLMTransientError,)`, `retry_backoff=True`, `max_retries=3-4`.
- Реализовать обработку финального провала: при исчерпании ретраев сохранять в БД сообщение с ролью `assistant` и текстом-заглушкой об ошибке, либо обновлять статус задачи так, чтобы клиент мог показать ошибку и кнопку "повторить".

## 2. Анализ снов нестабилен — зависает в статусе "в процессе"

**Симптом:** Пользователь записывает сон, анализ запускается автоматически, но индикатор "анализ в процессе" висит бесконечно. Анализ так и не приходит.

**Причина (корневая):** Апстрим Gonka Proxy нестабилен — сервер `85.234.91.100:8000` периодически недоступен (ConnectTimeout, ConnectError, i/o timeout). Это не баг в нашем коде, а проблема внешнего провайдера.

**Причина (усугубляющая — баг в нашем коде):** Когда Celery-задача `analyze_dream_task` исчерпывает все 4 ретрая по `LLMTransientError`, в обработчике ошибки статус анализа сбрасывается в `PENDING` (а не `FAILED`). После этого Celery больше не ретраит задачу, но анализ навсегда остаётся в `PENDING` — клиент продолжает показывать "анализ в процессе".

**Цепочка ретраев (текущая):**
1. Gonka Proxy: 3 попытки с фиксированной задержкой 5 сек (таймаут 120 сек на попытку)
2. LLM Service: 0 ретраев — просто пробрасывает 500
3. Backend llm_client: 0 ретраев — классифицирует ошибку как LLMTransientError
4. Celery: 4 ретрая с экспоненциальным бэкоффом (макс 120 сек между попытками)
5. **Итого в худшем случае:** до 12 запросов к Gonka Proxy, суммарное время ожидания может превышать 30 минут

**Дополнительная проблема:** При провале структурированного анализа (первый вызов LLM) происходит фоллбэк на текстовый анализ (второй вызов LLM). Каждый вызов проходит через все 3 ретрая Gonka Proxy. Это удваивает нагрузку на нестабильный апстрим и удваивает время ожидания.

**Решение:**
- Исправить обработку исчерпания ретраев: при финальном провале `LLMTransientError` статус анализа должен переходить в `FAILED`, а не оставаться в `PENDING`. Для этого нужно добавить обработчик `on_failure` в задачу или использовать `try/except` с проверкой `self.request.retries >= self.max_retries`.
- Добавить fallback-провайдер CometAPI (см. Issue 3).
- На клиенте: реализовать таймаут ожидания анализа с возможностью ручного повтора (кнопка "повторить анализ"), а не бесконечный спиннер.

## 3. Fallback-провайдер: CometAPI вместо Gonka Proxy

**Контекст:** Gonka Proxy (`proxy.gonka.gg`) проксирует запросы к апстриму `85.234.91.100:8000`, который периодически недоступен (ConnectTimeout, i/o timeout). Это корневая причина Issues 1 и 2. Нужен автоматический переход на резервного провайдера.

**Решение: автоматический fallback на CometAPI после 3 неудачных попыток Gonka Proxy.**

### Архитектура fallback

Переключение происходит на уровне LLM Service (`llm_service/`), прозрачно для бэкенда и Celery-задач.

**Логика:**
1. LLM Service пытается выполнить запрос через Gonka Proxy (основной провайдер, модель `Qwen/Qwen3-235B-A22B-Instruct-2507-FP8`).
2. Gonka Proxy выполняет до 3 попыток с задержкой 5 сек (существующая логика ретраев внутри `GonkaProxyProvider`).
3. Если все 3 попытки провалились (любая ошибка: таймаут, 5xx, connection refused) — LLM Service автоматически переключается на CometAPI и делает запрос туда.
4. Если CometAPI тоже не ответил — ошибка пробрасывается наверх в бэкенд как обычно.

### CometAPI — параметры подключения

- **Эндпоинт:** `POST https://api.cometapi.com/v1/chat/completions`
- **Авторизация:** `Authorization: Bearer {COMET_API_KEY}` (ключ уже есть в системе — используется для эмбеддингов и транскрипции)
- **Модель:** `gpt-5.1`
- **Формат запроса:** OpenAI-совместимый — `model`, `messages` (role + content), `temperature`
- **Формат ответа:** стандартный OpenAI Chat Completion — `choices[0].message.content`

### Реализация в LLM Service

**Новый провайдер: `llm_service/providers/comet_chat.py`**
- Класс `CometChatProvider` с тем же интерфейсом, что и `GonkaProxyProvider` (методы `analyze_dream`, `chat_completion`).
- Base URL: `https://api.cometapi.com/v1`
- Модель: `gpt-5.1`
- Таймаут: 120 сек (как у Gonka Proxy)
- Собственные ретраи: 2 попытки (CometAPI стабильнее, не нужно 3)

**Конфигурация: `llm_service/config.py`**
- Новые переменные окружения: `COMET_API_KEY` (ключ), `COMET_CHAT_MODEL` (модель, по умолчанию `gpt-5.1`)
- `COMET_API_KEY` уже существует в бэкенде (используется `embedding_service` и `transcription_service`), нужно пробросить его в контейнер `llm_service` через docker-compose.

**Оркестрация fallback: `llm_service/main.py`**
- В эндпоинтах `/analyze` и `/chat` обернуть вызов `llm_provider` (Gonka Proxy) в try/except.
- При любом исключении от Gonka Proxy — логировать предупреждение и повторить запрос через `comet_provider` (CometAPI).
- Логировать, какой провайдер в итоге ответил (`[Gonka]` / `[CometAPI fallback]`).

### Совместимость

CometAPI использует стандартный OpenAI Chat Completions формат. `GonkaProxyProvider` уже работает с этим же форматом (messages с role/content, ответ в `choices[0].message.content`). Поэтому `CometChatProvider` может повторить ту же логику нормализации сообщений и извлечения контента.

Разница в моделях (Qwen3-235B vs gpt-5.1) допустима: оба способны генерировать структурированный JSON-анализ по системному промпту. Качество анализа может отличаться, но пользователь получит результат вместо бесконечного спиннера.

### Что НЕ меняется

- Бэкенд (`llm_client.py`, `tasks.py`) не знает о fallback — для него LLM Service по-прежнему отвечает на `/analyze` и `/chat`.
- Celery-ретраи остаются как страховка на случай, если оба провайдера недоступны.
- Эмбеддинги и транскрипция продолжают работать напрямую через CometAPI из бэкенда — без изменений.
